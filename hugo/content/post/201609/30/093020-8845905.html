科學計算的程序編寫和通常所說的碼農的編程有多大的區別？

編程序，只不過是人類與計算機這種生產力工具對話的一種方式。

沒有計算機的時候，鐵匠靠手工技能加工金屬，後來靠操縱電動工具加工金屬；文官靠書信、紙質報表進行治理；會計靠籌算、珠算、計算器來算賬；科學家通過手工藝做實驗，通過書信發表文章，通過筆算／心算／籌算／計算器來進行數據分析。

現在有了計算機，編程讓數控機牀加工金屬；字處理工具、宏（廣義編程）、報表系統等來進行行政管理和財務管理；科學家可以用計算機仿真等手段進行一部分實驗工作，通過 Latex 等排版工具編寫文章，通過 Fortran、R、Matlab、Python、ROOT 等等語言進行數據分析。

計算機是很多行業中，面向人類開放的機器接口，而碼農（程序員）則是相應的行業中，面向機器開放的人類接口：直接從事生產的勞動者嘛。只不過，狹義的碼農（很多行業都有人自嘲爲碼農，但其實不是真正的碼農）其實就是除了編程，啥也不會的一個工種。這和其他專業領域首先具備專業技能，其次利用計算機加快生產的工種不同。

所以，其實，拿今天（利用計算機開展工作的）科學家和今天各行各業的碼農進行對比，就好像古典時期的科學家和鐵匠／文官／會計對比一樣，恐怕共同點僅僅是都有手藝／會寫字／會計算。但要求是不同的。

比如，爲了科學計算而編寫程序時，軟件生命週期、配置項的成熟度等等軟件工程方面考慮的東西，科學家往往不關心。但是對於參數的選取、對必要的近似的說明等等，則又是特別小心、必須明確的。因爲要拿出去跟其他的結果對比。

比如，我前幾天爲了畫下面這種圖：

模擬近地小天體的軌道、尺寸、反照率的分佈。如果只是爲了畫圖的話，大概其差不多就行了。但是因爲後面要對望遠鏡進行仿真，所以必須定好小天體的數密度隨尺寸的 broken power-law 的尺寸指數，並且用已經觀測到的近 1000 顆公里級小天體的樣本數量來歸一，等等。重要的參數，一定要註釋清楚，是從哪篇文章裏看來的。量綱一定要一致。

但至於說到我用來畫圖的程序本身，註釋是否清楚，是否可以從 python 2 移植到 python 3，我不關心。圖中太陽、水星、金星、地球、火星和木星的尺寸我卻又不關心了，因爲與估算望遠鏡本身的性能無關，僅僅起到“示意”作用而已，尺寸隨便取取就好了。另外，幾顆大行星的軌道位置都對嗎？不對。爲啥不對？方便畫箭頭啊！

上面是一個“糙快猛”的例子。但也不是一直這樣的。比如，前段時間，我需要解決一個球面上不具有旋轉對稱性的 Kernel 的卷積以及解卷積問題。目前現成的方法要麼需要假設 kernel 軸對稱，要麼需要對球面做平面近似。但因爲我要分析的是巡天數據，球面已經不可能近似成平面了，所以就想了別的辦法，找到一組正交的軸對稱基，把 kernel 分解，然後就可以快速計算了，還順便找到一個快速的矩陣 - 矩陣乘法算法。

所以，回到問題，如果說科學計算和通常所說的碼農的編程有多大區別，那麼區別還挺大的。因爲科學計算一般不受軟件工程的約束，一個研究課題的週期也比較短，少量科學計算軟件工具有可能變成某個領域、某個行業的通用工具，但大部分軟件都是隨開題生，隨結題亡。幾年前，我試圖想把我過去寫的程序再整理整理，挖掘一些可共用的庫出來，弄個工具包，但這個工作到現在也沒完成，更可怕的是，需要整理的代碼增加快，而整理的速度很慢——考慮框架等較爲頂層的設計，根本就不經濟。
